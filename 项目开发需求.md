# 个人工作面板 —— 需求文档 v5.0(定稿版)
> 本文档面向负责具体开发实现的 AI 编程智能体,可直接作为开发依据。字段定义、状态流转、交互规则均已明确,无遗留待确认事项;实现中若遇到本文未覆盖的细节,请按"最小化、不打扰用户"的原则自行决策。

---

## 一、项目背景与目标

**使用者**:系统开发工程师,制造业信息化/AI化项目负责人,个人单机使用
**核心定位**:常驻系统托盘、任务栏完全隐身的本地效率工具。以「项目空间」中的 Task(项目)为核心驱动单位,搭配「今日待办」工作会话视图与「灵感」速记模块

**明确不做**:
- 不做多用户协作、不做云端同步/跨设备访问
- 不做悬浮速记弹窗(该设计已废弃,由「小窗模式」统一承担)
- 数据存储对用户完全无感知,不暴露"数据库"概念

## 二、顶层导航结构

应用内统一提供**三个可切换的视图 Tab**,无论处于「主图模式」还是「小窗模式」,三个 Tab 始终存在:

```
┌───────────────────────────────────┐
│  [今日待办]   [灵感]   [项目空间]     │  ← 顶部/侧边导航,两种模式通用
├───────────────────────────────────┤
│                                     │
│           (当前 Tab 内容)             │
│                                     │
└───────────────────────────────────┘
```

### 两种窗口模式

| 模式 | 说明 | 默认打开 Tab |
|---|---|---|
| **主图模式** | 全屏/大窗口,用于深度整理与查看完整内容 | 记忆用户上次所处的 Tab |
| **小窗模式** | 紧凑型窗口,可半常驻于桌面角落,用于随手查看与快速操作 | **固定默认打开「今日待办」**,保证一呼出就能清晰看到当日待完成的 To-do 清单,不受上次使用状态影响 |

**窗口行为规则(关键)**:
- 应用**不在 Windows 任务栏显示图标**(需在窗口配置中设置 `skip_taskbar: true` 或等效属性),仅系统托盘图标可见
- 全局快捷键(如 `Ctrl+Shift+Space`)用于**呼出/隐藏**当前窗口
  - 若当前处于小窗模式:呼出时固定展示「今日待办」Tab(见上表)
  - 若当前处于主图模式:呼出时恢复用户上次所在 Tab
- 小窗模式与主图模式之间通过界面内的「展开/收起」按钮手动切换
- 小窗模式下「今日待办」的 To-do 清单需保证**完整可读**:窗口最小高度需能容纳至少 5-6 条待办项的可视区域,超出部分通过滚动查看,避免文字截断或过度压缩导致看不清
- 点击托盘图标 = 呼出窗口(等效于快捷键);托盘右键菜单提供「打开」「退出」

## 三、核心数据结构(Data Model)

时间戳统一使用 **ISO 8601 格式,精确到秒**。

### 3.1 Task(项目,归属「项目空间」)
```
Task {
  id: string (uuid)                // 内部唯一标识,不面向用户展示
  task_no: integer                 // 面向用户的项目序号,创建时自动分配,严格递增,
                                    // 即使项目被删除也不回收复用该号码
  name: string
  status: "active" | "completed"
  created_at: timestamp            // 创建时间,不可变
  last_progress_at: timestamp      // 最近推进时间,更新规则见下方
  completed_at: timestamp | null
  summary: string                  // 项目总结正文,支持多行文本,可为空
  summary_images: string[]         // 项目总结配图,本地文件路径列表,可为空
}
```
**`last_progress_at` 更新规则**:新增/编辑/删除/勾选该 Task 下任意 To-do 或 Issue 时自动更新为当前时间。单纯打开查看不算推进。编辑 `summary` 字段**也视为一次推进**,同样更新该时间戳。

### 3.2 To-do(待办项,归属 Task)
```
Todo {
  id: string (uuid)
  todo_no: integer                 // 面向用户的序号,在其所属 Task 内部严格递增,不回收复用
  task_id: string
  content: string
  images: string[]
  status: "open" | "done"
  created_at: timestamp
  completed_at: timestamp | null
  order: number                    // Task 详情页内手动拖拽排序用
}
```

### 3.3 Issue(问题记录,归属 Task)
```
Issue {
  id: string (uuid)
  task_id: string
  title: string
  detail: string
  images: string[]
  status: "open" | "resolved"
  created_at: timestamp
  resolved_at: timestamp | null
}
```

### 3.4 Idea(灵感,独立实体)
```
Idea {
  id: string (uuid)
  content: string
  images: string[]
  created_at: timestamp
}
```

### 3.5 今日待办会话状态(轻量瞬态,不做持久化实体)
「今日待办」本质是一个**实时查询视图 + 一个瞬态会话状态**,不新增持久化数据表:
```
TodaySession (应用运行时内存状态,不写入长期存储) {
  selected_task_ids: string[]      // 当前被拉入"今日待办"的 Task id 列表(可多选)
}
```
- 用户在「项目空间」选择某个 Task 加入今日待办 → 将该 `task_id` 加入 `selected_task_ids`
- 「今日待办」视图渲染逻辑 = 遍历 `selected_task_ids`,对每个 Task **实时查询**其 `status = open` 的 Todo,按 Task 分组展示
- 新增 To-do 只能在 Task 详情页(项目空间内)进行(见模块 D)。只要该 Task 当前在会话中,新增的 To-do 会因为实时查询机制**自动出现**在今日待办分组里,无需任何额外操作
- 用户在今日待办视图中勾选完成某条 To-do → 直接更新该 Todo 的 `status = done`、写入 `completed_at`(操作真实数据,非副本),该条目立即置灰但暂留在当日视图中,方便回顾
- 点击「结束今天的工作」→ 展示总结界面(见模块 F)→ 确认后清空 `selected_task_ids`(会话状态归零),今日待办视图恢复空白
  - 未完成 To-do 因从未被"移动"过,本来就还在原 Task 的 To-do list 里,无需归还操作
  - 已完成 To-do 的 `status = done` 已是真实持久化状态,下次该 Task 再次被加入今日待办时,查询条件 `status = open` 自动排除已完成项

## 四、功能需求(按模块 MECE 拆解)

### 模块 A:今日待办(Today Session)
- A1. 视图顶部提供「+ 添加项目」入口 → 弹出项目选择器(列出「项目空间」中所有 `status = active` 的 Task,按 `task_no` 显示,支持多选)
- A2. 选中的每个 Task 在今日待办视图中形成一个分组:
  ```
  【No.03 MES系统对接】                    ← 分组标题,字号较小
    01. 完成接口联调文档                    ← To-do 内容,字号较大,视觉主体
    02. 与产线沟通字段定义
    [x] 03. 确认数据库表结构(已完成,置灰)

  【No.07 AI质检算法验证】
    01. 采集测试样本图片
  ```
  - 分组标题展示该 Task 的 `task_no` 与名称,字号**小于** To-do 正文字号(To-do 是视觉主体)
  - 每条 To-do 前展示其 `todo_no`
- A3. 支持在当前视图直接勾选完成 / 取消完成(实时写入 Todo 真实数据)
- A4. **不支持**在今日待办视图内直接新增 To-do;新增操作统一在 Task 详情页完成(见模块 D),已在会话中的 Task 新增后会自动同步显示于此
- A5. 顶部/底部提供「结束今天的工作」按钮,点击后展示总结界面(模块 F),确认关闭后清空会话状态

### 模块 B:灵感(Idea)速记
- B1. 独立 Tab,时间倒序列表展示
- B2. 支持文字 + 粘贴图片(`Ctrl+V`)
- B3. 增删改
- B4.(可选后期功能)一键转化为某 Task 下的 To-do

### 模块 C:项目空间(统一的项目管理区,含进行中与已完成)
- C1. 顶部展示统计信息:**「未完成项目数」/「已完成项目数」**(实时统计)
- C2. 提供筛选切换:「进行中」/「已完成」/「全部」
- C3. 卡片网格展示 Task,每张卡片展示:`task_no`、项目名称、`last_progress_at`(进行中卡片)或 `completed_at`(已完成卡片)、未完成 To-do 数、未解决 Issue 数
  - **已完成卡片额外展示 `summary` 的前 30-40 字摘要预览**(若为空则不展示该行),方便用户不点开详情就能快速回忆项目成果
- C4. 已完成 Task 卡片应用模块 E3 的暗淡视觉规则
- C5. 新建 Task(仅需输入名称,系统自动分配 `task_no`,写入 `created_at`/`last_progress_at`)
- C6. 点击「进行中」卡片进入可编辑详情页(模块 D);点击「已完成」卡片进入只读详情页(**「项目总结」字段例外,见模块 G,即使项目已完成仍可编辑**)
- C7. **「标记完成」操作流程(重要,与模块 G 联动)**:
  1. 用户点击「标记完成」按钮
  2. 弹出「项目总结」编辑框(若 `summary` 已有内容则预填充,供用户在此基础上补充修改)
  3. 用户填写/确认总结内容(支持文字 + 粘贴图片)后点击「确认完成」
  4. 系统写入 `summary` / `summary_images`,并设置 `status = completed`、`completed_at = 当前时间`
  5. 若用户在弹窗中选择「暂不填写,直接完成」,系统允许跳过(不做强制阻塞),但默认按钮态引导用户优先填写
- C8. Task 删除(需二次确认)
- C9. 已完成 Task 提供「导出为 Markdown」按钮:
  - 导出内容:项目名称、`task_no`、`created_at`、`last_progress_at`、`completed_at`、**项目总结(`summary` 及配图)**、完整 To-do list(含 `todo_no` 与完成状态)、完整 Issue list(含状态与详情描述)
  - 导出结构建议:总结章节置于文档顶部(紧跟基础信息之后),便于日后快速翻阅回顾成果,再往下依次是 To-do 与 Issue 明细
  - 图片资源一并复制到与 `.md` 文件同名的资源文件夹,Markdown 内用相对路径引用
  - 默认导出路径:程序目录 `/exports/`,文件命名 `{task_no}_{项目名称}_{completed_at日期}.md`
  - (可选)支持批量导出全部已完成 Task

### 模块 D:Task 详情页 —— 左右布局
```
┌─────────────────────┬─────────────────────┐
│   左栏:To-do List    │   右栏:Issue List     │
│  01. [ ] 待办事项      │  ▸ 问题标题 1          │
│  02. [x] 待办事项(暗)│  ▸ 问题标题 2(点击展开)│
│  [+ 新增待办]          │  [+ 新增问题]          │
└─────────────────────┴─────────────────────┘
│         下方:项目总结编辑区(模块 G,常驻可展开)     │
└───────────────────────────────────────────┘
```
- D1. 左栏:To-do 完整增删改查、拖拽排序、勾选态(应用模块 E1 视觉规则)、每条展示 `todo_no`。**这是新增 To-do 的唯一入口**
- D2. 右栏:Issue 列表,默认只显示标题,点击展开为详情编辑区(问题现象/临时方案等,支持多行文字+图片)
- D3. 两栏独立滚动;窗口宽度不足时可上下堆叠(小窗模式下需考虑此响应式规则)
- D4. 详情页顶部展示 `task_no`、`created_at`、`last_progress_at`(悬停显示精确到秒的完整时间戳)
- D5. 详情页底部或侧边固定展示「项目总结」编辑区入口(见模块 G),进行中与已完成状态下均可访问

### 模块 E:视觉状态规则(全局通用)
- E1. To-do 勾选完成:整体 `opacity: 0.45`,文字颜色降为低饱和度灰蓝(`#8A9BB5` 左右),不加删除线
- E2. Issue 标记已解决:同上视觉规则,标题旁可加小标签「已解决」
- E3. Task 标记完成(项目空间已完成卡片):整卡应用同等暗淡处理
- E4. 以上规则在「今日待办」视图中同样适用(已完成 To-do 置灰但暂留在视图内,直到「结束今天的工作」清空会话)

### 模块 F:今日工作完成总结界面
- F1. 触发时机:点击「结束今天的工作」后,清空会话前展示
- F2. **内容极简**:仅需醒目展示 **「今日工作已完成 ✔️」**,不需要额外的统计数字或分项罗列
- F3. 视觉:延续深色藏青毛玻璃风格,可用较大字号 + 主强调色勾选图标(✔️)居中展示,营造干净利落的完成感
- F4. 用户点击「确认/关闭」后才真正清空会话状态
- **注**:此为「今日工作」会话层面的完成提示,与模块 G「项目总结」(Task 层面的成果记录)是两个独立概念,不要混淆实现

### 模块 G:项目总结(新增)★
- G1. 每个 Task 拥有一个 `summary` 文本区(支持多行)+ `summary_images` 配图区,在 Task 详情页内**常驻可见**(建议做成可展开/收起的卡片区块,默认收起以节省空间,点击展开编辑)
- G2. 进行中状态下,用户可随时编辑总结内容(比如推进过程中就先写点阶段性心得,不必等到项目彻底结束)
- G3. 触发「标记完成」时,自动弹出总结编辑框强化提醒(见模块 C7 流程),但不阻塞已有内容的编辑权限
- G4. 已完成状态下,Task 详情页整体转为只读,**唯独 `summary` / `summary_images` 保持可编辑**,方便日后回顾时补充或修正总结内容
- G5. 编辑 `summary` 视为一次"推进"操作,更新 `last_progress_at`(见 3.1 节字段说明)
- G6. 已完成 Task 的项目空间卡片展示 `summary` 摘要预览(见 C3)
- G7. Markdown 导出时,总结内容置于文档顶部章节(见 C9)

## 五、UI / 视觉设计规范

**整体风格**:深色系毛玻璃(Acrylic / Glassmorphism),藏青蓝主色调

| 元素 | 规范 |
|---|---|
| 背景 | 深藏青渐变(`#0F1B2D` → `#16223A`)+ `backdrop-filter: blur()` |
| 主强调色 | 亮蓝 `#4FA3F7` / 天青蓝 `#3E8EDE` |
| Task 卡片(进行中) | `rgba(30, 45, 70, 0.6)` + 1px 半透明蓝边框 + 轻微投影 |
| Task 卡片(已完成) | 同上叠加 `opacity: 0.5` |
| 已完成条目(To-do/Issue) | `opacity: 0.45` + 文字色 `#8A9BB5` 左右,不加删除线 |
| 今日待办分组标题 | 字号小于 To-do 正文,颜色可用主强调色的低饱和版本以区分层级 |
| 项目总结区块 | 建议使用与主体略有区分的卡片底色(如更深一层的藏青 `rgba(20, 32, 52, 0.7)`),突出"沉淀内容"的独立感 |
| 中文字体 | PingFang SC / Microsoft YaHei UI |
| 英文数字字体 | Inter 或 SF Pro |
| 图标风格 | 线性图标(Outline),克制专业 |
| 动效 | 状态切换用透明度渐变(150-200ms ease) |

## 六、技术架构建议

- **框架**:Tauri(桌面壳,需配置 `skip_taskbar: true`)+ React 或 Vue
- **本地数据存储**:结构化本地文件(SQLite 或 JSON 均可,由实现方判断,原则"轻量、用户无感知")
- **序号生成**:`task_no` 与 `todo_no`(按所属 Task 独立计数)均采用持久化自增计数器,不因删除而回收复用
- **今日待办**:纯前端/内存状态管理会话选中的 `task_id` 列表,不新增持久化表,数据操作均直接作用于 Todo 真实记录
- **图片处理**:剪贴板监听(小窗模式、详情页编辑区、项目总结编辑区均需支持)→ 本地图片目录保存 → 缩略图渲染
- **Markdown 导出**:序列化 Task 完整结构(含总结)为 Markdown,处理图片资源复制与相对路径替换

## 七、非功能性需求

- 快捷键呼出/隐藏窗口响应时间目标:200ms 内
- 任务栏零露出,仅托盘图标可见
- 常驻托盘状态下内存/CPU 占用需控制在低水平
- 数据写入需保证异常关闭不丢失最近编辑内容
- 「项目空间」统计数字(未完成/已完成项目数)需实时准确
- 小窗模式默认展示今日待办且需保证清单可读性(见第二节窗口行为规则)

## 九、需求边界说明(供实现方参考,非待确认项)

以下为已经明确决策、无需再向用户确认的设计选择,实现方按此执行即可:
- 今日待办视图内不提供"新增 To-do"入口,统一通过 Task 详情页操作,依赖实时查询自动同步
- 小窗模式呼出时固定展示「今日待办」Tab,不跟随主图模式的"记忆上次 Tab"逻辑
